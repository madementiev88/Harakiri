## 1. Обзор проекта

### 1.1 Цель и задачи

Разработка Telegram Mini App для барбершопа HARAKIRI — это современный цифровой инструмент самозаписи, интегрированный непосредственно в экосистему Telegram. Приложение позволяет клиентам без перехода на сторонний сайт найти мастера, выбрать одну или несколько услуг, удобное время записи, а также отменить или перенести запись.

**Бизнес-цели:**

- Снижение нагрузки на администраторов (автоматизация записи)
- Привлечение лояльной аудитории через удобный мобильный интерфейс
- Перемещение воронки продаж через Telegram-канал

**Технические цели:**

- Глубокая интеграция с YCLIENTS для синхронизации расписания в реальном времени
- Zero-friction UX: запись за 3 шага без регистрации
- Telegram-авторизация (без паролей и SMS)
- Адаптивный интерфейс для всех устройств (iOS / Android)
- Корзина услуг — возможность комбинировать несколько услуг в одной записи

### 1.2 Целевая аудитория

| Аудитория | Описание |
|-----------|----------|
| Клиенты барбершопа | Мужчины 18-45 лет, активные пользователи Telegram, ценят скорость и удобство |
| Администраторы | Управляют расписанием и записями через YCLIENTS, получают уведомления |

### 1.3 Ключевые сценарии использования

1. Клиент открывает Mini App в Telegram-боте барбершопа
2. Просматривает список мастеров
3. Выбирает мастера → одну или несколько услуг → дату и время
4. Подтверждает запись (данные из Telegram-профиля)
5. Получает подтверждение и напоминания за 24ч и 2ч
6. По необходимости — отменяет или переносит запись

## 3. Интеграция с YCLIENTS

### 3.1 Параметры подключения

| Параметр | Значение |
|----------|----------|
| Salon ID | 1178599 |
| Base URL | `https://api.yclients.com/api/v1` |
| Auth URL | `https://n1178599.yclients.com/` |

Для работы с YCLIENTS API необходимы два токена: партнёрский токен (Partner Token) для идентификации приложения и пользовательский токен (User Token) для выполнения операций от имени администратора.

### 3.2 Важные ограничения и риски YCLIENTS API

> **Этот раздел — обязательное чтение перед началом разработки.**

**Требуется проверить до начала Фазы 1:**

1. **Мульти-услуги в одной записи** — необходимо протестировать, принимает ли `POST /records` массив из нескольких `service_id` в поле `appointments[].services`. Если нет — потребуется создавать последовательные записи или отправлять каждую услугу отдельным appointment. Это критически влияет на архитектуру корзины.

2. **Rate limits API** — YCLIENTS ограничивает количество запросов (обычно 100-200 запросов/мин на токен). При 50+ одновременных пользователях с кэшем слотов 60 сек можно превысить лимит. Необходимо реализовать очередь запросов на бэкенде и мониторить заголовки `X-RateLimit-*`.

3. **Срок жизни User Token** — YCLIENTS User Token может протухнуть. Бэкенд должен:
   - Перехватывать 401-ответы от YCLIENTS
   - Автоматически запрашивать новый токен через `POST /auth`
   - Повторять исходный запрос с новым токеном
   - Логировать каждый случай обновления токена

4. **Формат ответов API** — все примеры ответов в этом документе упрощены. Реальные ответы YCLIENTS оборачивают данные в `{"success": true, "data": {...}, "meta": {...}}`. Обработка ошибок: `{"success": false, "meta": {"message": "..."}}`.

### 3.5 Получение доступных слотов

Ключевой запрос для экрана выбора времени. Возвращает массив свободных временных слотов для конкретного мастера на указанную дату с учётом длительности выбранных услуг:

```http
GET /api/v1/timetable/{salon_id}?staff_id={staff_id}&date={YYYY-MM-DD}&service_id={service_id}
```

> **Корзина услуг:** при выборе нескольких услуг необходимо передавать суммарную длительность для корректного расчёта доступных слотов. Если YCLIENTS API не поддерживает передачу нескольких service_id в одном запросе, бэкенд должен рассчитать суммарную длительность и отфильтровать слоты, в которые помещается вся корзина.

Пример ответа (упрощённо):

```json
{
  "data": [
    {"time": "10:00", "seance_length": 60, "staff": {"id": 123}},
    {"time": "11:00", "seance_length": 60, "staff": {"id": 123}}
  ]
}
```

### 3.6 Создание записи

```http
POST /api/v1/records/{salon_id}
```

```json
{
  "phone": "+7XXXXXXXXXX",
  "fullname": "Имя клиента",
  "email": "",
  "type": 1,
  "comment": "Запись через Telegram Mini App",
  "appointments": [
    {
      "id": 1,
      "services": [service_id_1, service_id_2],
      "staff_id": 456,
      "datetime": "2026-03-15 14:00:00"
    }

```

> **Мульти-услуги:** поле `services` принимает массив ID услуг. При записи на несколько услуг (например, стрижка + борода) все ID передаются в одном массиве. YCLIENTS автоматически рассчитает суммарную длительность и стоимость. Бэкенд должен валидировать, что выбранный слот вмещает все услуги.

### 3.7 Стратегия кэширования

| Данные | Стратегия кэширования |
|--------|----------------------|
| Список мастеров | TTL: 30 минут (редко меняется) |
| Список услуг + цены | TTL: 30 минут |
| Расписание (рабочие дни) | TTL: 1 час |
| Доступные слоты | TTL: 60 секунд (часто меняется) |
| Записи клиента | TTL: 10 минут |

### 3.8 Таймзона

> Все даты и время в системе используют таймзону **Europe/Moscow (UTC+3)**.

- Бэкенд: переменная окружения `TZ=Europe/Moscow`, все cron-задачи работают в этой зоне
- База данных: все TIMESTAMP-поля хранятся как `TIMESTAMPTZ`, конвертируются при записи
- YCLIENTS API: возвращает время в локальной таймзоне салона (Москва) — дополнительная конвертация не требуется
- Фронтенд: отображает время как получено от бэкенда (уже в Europe/Moscow)

## 5. Жизненный цикл записи (State Machine)

### 5.1 Диаграмма состояний

```
                    ┌──────────────────────────────────────────┐
                    │                                          │
  POST /bookings    ▼                                          │
 ─────────────▶ [pending] ──── sync cron "confirmed" ──▶ [confirmed]
                    │                                     │
                    │                                     │
                 sync cron     отмена клиентом         cron: время
               "cancelled"    DELETE /bookings/:id     визита + duration
                    │              │                      │
                    ▼              ▼                      ▼
               [cancelled]    [cancelled]           [completed]
```

### 5.2 Переходы состояний

| Из | В | Триггер | Кто инициирует |
|----|---|---------|---------------|
| — | pending | Клиент создаёт запись через Mini App | POST /api/bookings |
| pending | confirmed | Cron `sync_bookings` обнаружил подтверждение в YCLIENTS | Автоматически |
| pending | cancelled | Клиент отменяет или cron `sync_bookings` обнаружил отмену | DELETE /api/bookings/:id или cron |
| confirmed | cancelled | Клиент отменяет (не позднее 2ч до визита) или администратор | DELETE /api/bookings/:id или cron |
| confirmed | completed | Cron-задача: `booking_date + booking_time + total_duration < NOW()` | Автоматически (cron `complete_bookings`, каждые 15 мин) |
| completed | — | Финальное состояние | — |
| cancelled | — | Финальное состояние | — |

### 5.3 Механизм перевода в `completed`

Cron-задача `complete_bookings` (каждые 15 минут):

```sql
UPDATE bookings_log
SET status = 'completed'
WHERE status = 'confirmed'
  AND (booking_date + booking_time + (total_duration || ' minutes')::INTERVAL) < NOW()
  AND booking_date >= CURRENT_DATE - INTERVAL '2 days';
```

### 5.4 Синхронизация с YCLIENTS

Cron-задача `sync_bookings` (каждые 30 мин) обеспечивает синхронизацию статусов:

- Делает `GET /records/{salon_id}` и сверяет статусы
- При расхождении — обновляет локальный статус и уведомляет клиента
- Логирует каждое расхождение для мониторинга

### 5.5 Перенос записи

Перенос записи реализуется как **отмена текущей + создание новой**:

1. Клиент нажимает "Перенести" на активной записи (экран 5)
2. Текущая запись отменяется (DELETE на YCLIENTS + status → cancelled)
3. Открывается экран 2 с предзаполненным мастером и услугами
4. Клиент выбирает новую дату/время и подтверждает
5. В `bookings_log` создаётся новая запись, в поле `comment` добавляется: "Перенос записи #{old_id}"

> Перенос доступен не позднее чем за 2 часа до визита (как и отмена).

### 6.3 Авторизация пользователя

Telegram передаёт данные пользователя через initData — строка, подписанная HMAC-SHA256. Бэкенд обязательно проверяет подпись для безопасной идентификации пользователя.

**Стратегия авторизации (initData vs JWT):**

- **initData** — используется при каждом открытии Mini App для первичной аутентификации. Фронтенд отправляет initData на `POST /api/auth`, бэкенд верифицирует подпись и выдаёт JWT.
- **JWT** — используется для всех последующих API-запросов в рамках сессии. Срок жизни: 24 часа. Хранится в памяти приложения (не в localStorage).
- **При протухании JWT** — фронтенд повторно отправляет initData на `/api/auth` (молча, без перезагрузки UI).
- **Cron-задачи и бот** — не используют ни initData, ни JWT. Работают с серверными токенами YCLIENTS напрямую.

```javascript
// Frontend: получение данных пользователя
const user = tg.initDataUnsafe?.user;
const initData = tg.initData; // Отправляем на бэкенд для верификации

// Backend: верификация initData
const crypto = require('crypto');

function verifyTelegramData(initData, botToken) {
  const data = new URLSearchParams(initData);
  const hash = data.get('hash');
  data.delete('hash');

  const dataCheckString = [...data.entries()]
    .sort()
    .map(([k, v]) => `${k}=${v}`)
    .join('\n');

  const secretKey = crypto
    .createHmac('sha256', 'WebAppData')
    .update(botToken)
    .digest();

  const expectedHash = crypto
    .createHmac('sha256', secretKey)
    .update(dataCheckString)
    .digest('hex');

  return expectedHash === hash;
}
```

### 6.4 Экраны и пользовательский флоу

#### Экран 1: Главный экран (выбор мастера)

- Логотип и подзаголовок HARAKIRI
- Grid-сетка карточек мастеров (фото, имя, специализация)
- Кнопка **"Любой мастер"** — запись без привязки к конкретному мастеру:
  - Бэкенд выбирает **наименее загруженного мастера** на выбранную дату/время
  - На экране подтверждения (экран 4) клиент видит назначенного мастера **до** подтверждения
  - Если клиент не согласен — может вернуться и выбрать конкретного мастера
- Фильтр-переключатели или список с фильтрами

#### Экран 2: Выбор услуг (корзина)

- Аккордеон-список категорий (стрижки, борода, комплексы, уход)
- Каждая услуга: название, длительность, цена
- **Множественный выбор услуг** — чекбоксы/переключатели на каждой услуге
- **Плавающая панель корзины** внизу экрана:
  - Количество выбранных услуг
  - Суммарная длительность (например, "1ч 30мин")
  - Итоговая цена
  - Кнопка "Далее" (активна при выборе >= 1 услуги)
- При добавлении/удалении услуги — Haptic Feedback и анимация обновления корзины

#### Экран 3: Выбор даты и времени

- Горизонтально пролистываемый календарь (14 дней вперёд)
- Недоступные даты — выделены серым
- После выбора даты — grid свободных временных слотов
- Занятые слоты — недоступны и серые
- **Слоты фильтруются с учётом суммарной длительности корзины** — если корзина 90 мин, слоты, где остаётся меньше 90 мин до конца рабочего дня или следующей записи — недоступны

#### Экран 4: Подтверждение записи

- Сводка: мастер, **список всех выбранных услуг**, дата/время, **итоговая цена**
- Имя клиента (из Telegram) и телефон (ввод при первой записи)
- Чекбокс согласия на обработку персональных данных (ФЗ-152) со ссылкой на Политику конфиденциальности
- Кнопка-слайдер "Подтвердить запись" (Telegram MainButton)

#### Экран 5: Успешная запись

- Анимация успеха
- Детали записи с карточкой
- Переход к "Мои записи"

#### Экран 6: Мои записи

- Список активных записей с кнопками **"Перенести"** и **"Отменить"**
  - Перенос: отменяет текущую → открывает экран 2 с пресетом мастера и услуг → выбор новой даты
  - Обе кнопки доступны не позднее 2ч до визита
- Архив прошедших визитов

## 9. Безопасность

### 9.1 Ключевые требования безопасности

- Всегда валидировать Telegram initData на бэкенде (HMAC-SHA256)
- Никогда не выставлять секретные ключи на фронтенде без верификации
- YCLIENTS User Token хранить только на сервере (не в клиентском коде)
- Все запросы к YCLIENTS проксировать через собственный бэкенд
- Rate limiting: максимум 10 запросов/минуту на пользователя
- HTTPS обязательно (Telegram не открывает HTTP-приложения)
- Валидация всех входных данных (sanitize phone, name)
- Ограничение отмены записей: не позднее чем за 2 часа до визита

### 9.2 Персональные данные (ФЗ-152)

Приложение обрабатывает персональные данные пользователей (имя, фамилия, телефон, история записей). Необходимо:

1. **Согласие на обработку ПД** — чекбокс на экране подтверждения записи (экран 4), обязательный для первой записи
2. **Политика конфиденциальности** — публичный документ, доступный по ссылке из Mini App
3. **Минимизация данных** — собираем только необходимое: telegram_id, имя, телефон
4. Поля `pd_consent` и `pd_consent_date` в таблице `users` фиксируют факт согласия

### 9.4 Логирование

**Формат логов:** structured JSON:

```json
{
  "timestamp": "2026-03-15T14:00:00.000Z",
  "level": "info",
  "module": "booking",
  "action": "create_booking",
  "telegram_id": 123456789,
  "yclients_record_id": 12345,
  "duration_ms": 450,
  "trace_id": "abc-123-def"
}
```

**Уровни логирования:**

| Уровень | Что логируется |
|---------|---------------|
| error | Ошибки YCLIENTS API, ошибки создания записи, крэши |
| warn | Rate limit приближается, User Token обновлён |
| info | Запись создана/отменена, уведомление отправлено |
| debug | Тела запросов/ответов YCLIENTS (только в dev), кэш hit/miss |

**Трейсинг запросов:**

Каждый запрос от фронтенда получает `trace_id` (генерируется middleware). Этот ID передаётся во все внутренние вызовы и запросы к YCLIENTS. Позволяет отследить полный путь: фронт → бэкенд → YCLIENTS → уведомление.

- Время от кнопки "Записаться" до создания записи < 3 секунды
- Корректная работа на iOS 15+ и Android 9+ в актуальном Telegram
